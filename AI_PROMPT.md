# System Prompt для MSCodeBase Intelligence

## Как работает подключение в Zed

### Установка
1. Запусти `python install.py` — расширение установится в `%LOCALAPPDATA%\Zed\extensions\mscodebase-intelligence`
2. В настройки Zed автоматически добавятся:
   - MCP-сервер `mscodebase-intelligence` (для AI-ассистента)
   - LSP-сервер `mscodebase-lsp` (для проактивной индексации)
   - Системные правила в `agent.system_prompt`
3. При первом открытии проекта запусти LM Studio (для эмбеддингов)

### Запуск
- **MCP-сервер** запускается при старте Zed и обрабатывает запросы AI-ассистента
- **LSP-сервер** запускается при открытии проекта и реагирует на `Ctrl+S`
- При сохранении файла LSP автоматически индексирует его в фоне
- MCP сразу использует свежие эмбеддинги из общей БД

### Гибридная архитектура (LSP + MCP)
- **LSP-сервер** (`src/lsp_main.py`) — проактивно индексирует файлы при сохранении
- **MCP-сервер** (`src/main.py`) — реактивно отвечает на запросы AI-ассистента
- Оба сервера используют одну БД — свежие эмбеддинги всегда доступны

## Когда использовать MSCodeBase Intelligence

Используй этот инструмент, когда:

1. **Пользователь просит найти что-то в коде** — поиск функций, классов, методов, переменных
2. **Нужно понять архитектуру проекта** — структура файлов, связи между компонентами
3. **Требуется найти использование символа** — где вызывается функция, где используется класс
4. **Нужно собрать контекст по теме** — несколько файлов, связанные с конкретной задачей
5. **Пользователь спрашивает о коде** — объяснение, рефакторинг, анализ

## Как использовать инструменты

### 1. `index_project_dir` — Индексация проекта
**Когда**: При первом открытии проекта или после значимых изменений
**Как**: Запусти сразу после открытия проекта
```
index_project_dir(path="/путь/к/проекту")
```
**Важно**: Индексация работает асинхронно — результат приходит мгновенно, а сам процесс идёт в фоне.

### 2. `search_code` — Гибридный поиск по коду
**Когда**: Нужно найти конкретный код, функцию, класс
**Как**: Используй естественный язык
```
search_code(query="функция авторизации пользователя")
search_code(query="обработка ошибок API")
search_code(query="класс DatabaseManager")
```

### 3. `get_context` — Сбор контекста
**Когда**: Нужно собрать релевантные фрагменты кода для ответа
**Как**: Задавай конкретный запрос
```
get_context(query="логика подключения к базе данных")
get_context(query="обработка JWT токенов")
```

### 4. `get_symbol_info` — Информация о символе
**Когда**: Нужно найти определение и все использования конкретного символа
**Как**: Указывай точное имя символа
```
get_symbol_info(query="authenticateUser")
get_symbol_info(query="DatabaseManager")
```

### 5. `get_repo_map` — Карта репозитория
**Когда**: Нужно понять общую структуру проекта
**Как**: Передай корень проекта
```
get_repo_map(project_root="/путь/к/проекту")
```

### 6. `get_index_status` — Статус индекса
**Когда**: Хочешь проверить, проиндексирован ли проект
**Как**: Без аргументов
```
get_index_status()
```

### 7. `scan_changes` — Архитектурный дифф
**Когда**: Файлы изменены вне Zed (git pull, git checkout)
**Как**: Передай корень проекта
```
scan_changes(project_root="/путь/к/проекту")
```

### 8. `watcher_status` — Статус компонентов
**Когда**: Проверка здоровья системы
**Как**: Без аргументов
```
watcher_status()
```

## Порядок действий при работе с новым проектом

1. **Сначала** — `get_index_status()` чтобы проверить текущее состояние
2. **Если база пуста** — `index_project_dir(path=путь_к_проекту)`
3. **Потом** — используй `search_code`, `get_context`, `get_symbol_info` для анализа
4. **После изменений** — `scan_changes` для отслеживания

## Лучшие практики

- **Используй естественный язык** в поиске — "функция для валидации email" работает лучше, чем "validateEmail"
- **Комбинируй инструменты** — сначала найди символ через `search_code`, потом получи детали через `get_symbol_info`
- **Для рефакторинга** — используй `get_symbol_info` для поиска всех зависимостей перед изменением
- **Контекстный бюджет** — читай файлы по 50 строк, не целиком
- **Пути** — нормализуй в POSIX lowercase: `path.as_posix().lower()`

## Примеры сценариев

### Сценарий: "Найди все места, где используется класс User"
```
1. get_symbol_info(query="User")
2. Если нужно больше контекста — search_code(query="класс User")
```

### Сценарий: "Объясни, как работает авторизация в этом проекте"
```
1. get_context(query="авторизация пользователя login")
2. get_symbol_info(query="auth") или get_symbol_info(query="login")
```

### Сценарий: "Покажи структуру проекта"
```
1. get_repo_map(project_root="/путь/к/проекту")
```

### Сценарий: "Нужно найти баг в обработке ошибок"
```
1. search_code(query="обработка ошибок exception error handling")
2. get_context(query="error handling try catch")
```

## Ограничения

- Максимальный размер файла: 1 MB
- Файлы из `.gitignore` игнорируются
- Директории `.git`, `node_modules`, `venv`, `__pycache__` автоматически пропускаются
- Требуется запущенный LM Studio для эмбеддингов (или Ollama/ONNX fallback)
- Поддерживаемые языки: Python, Rust, TypeScript, JavaScript, Go
