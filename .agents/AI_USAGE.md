# MSCodeBase Intelligence — Инструкция для AI-ассистента

## Философия выбора инструментов

Zed предпочитает встроенные инструменты (минимальная задержка).
Используй MCP-инструменты только когда:
- Нужен семантический поиск (а не просто текст)
- Нужен граф вызовов / зависимостей
- Нужна история коммитов / анализ багов
- Нужен структурный поиск (AST)

Для простого чтения файлов и текстового поиска — используй встроенные инструменты Zed.

## Алгоритм принятия решения

### Шаг 1: Проверка индекса
```
get_index_status()
```
- Если `total_chunks == 0` → индекс пуст → используй `grep`/`find_path`
- Если `chunks > 0` → можно использовать семантический поиск

### Шаг 2: Выбор инструмента

**Используй MCP (медленнее, но мощнее):**
- `search_code` — семантический поиск (понимает смысл)
- `get_symbol_info` — определение + все вызовы символа
- `impact_analysis` — что сломается при изменении
- `get_file_history` — история изменений файла
- `get_bug_correlation` — горячие точки (часто ломающиеся файлы)
- `structural_search` — поиск по паттернам AST
- `cross_repo_search` — поиск по связанным проектам

**Используй встроенные (быстрее):**
- `grep` — точный текстовый поиск
- `find_path` — поиск файлов по маске
- `read_file` — чтение файла (до 50 строк)
- `terminal` — ad-hoc команды

## Инструменты MCP-сервера

| Инструмент | Назначение | Приоритет |
|------------|------------|----------|
| `search_code(query, mode=...)` | Семантический поиск (mode: fast/quality/deep/context/auto) | 🔥 Высокий |
| `get_symbol_info(name)` | Определение + вызовы | 🔥 Высокий |
| `impact_analysis(name)` | Анализ зависимостей | 🔥 Высокий |
| `get_file_history(path)` | История файла | 🔥 Высокий |
| `get_bug_correlation(path)` | Горячие точки | 🔥 Высокий |
| `structural_search(pattern)` | AST-паттерны | ⭐ Средний |
| `cross_repo_search(query)` | Мульти-репо поиск | ⭐ Средний |
| `get_repo_map()` | Карта проекта | ⭐ Средний |
| `get_index_status()` | Статус индекса | ✅ Обязательно |
| `get_logs()` | Логи проекта | ✅ Обязательно |

## Важные ограничения

- **Context budget:** Максимум 50 строк за `read_file`
- **State-awareness:** Всегда проверяй `get_index_status()` перед поиском
- **Безопасность:** Перед правкой — перечитай строки (`read_file`)
- **Окна:** Пути в MCP — нативные Windows (обратные слеши)
- **Синхронизация:** После `write_file` → `notify_change` → `get_index_status`

## Запреты

- ❌ Не используй MCP для простого чтения файлов (read_file быстрее)
- ❌ Не используй MCP для точного текстового поиска (grep быстрее)
- ❌ Не угадывай номера строк (используй get_symbol_info)
- ❌ Не читай файлы > 50 строк целиком
- ❌ Не игнорируй пустой индекс (chunks == 0)

## Пример рабочего процесса

```
1. Пользователь: «Найди все места, где используется функция auth»
   → Используем: get_symbol_info("auth")

2. Пользователь: «Почему этот модуль часто ломается?»
   → Используем: get_bug_correlation("src/module.py")

3. Пользователь: «Найди все вызовы БД»
   → Используем: search_code("database query connection")

4. Пользователь: «Прочитай файл config.py»
   → Используем: read_file (встроенный, быстрый)
```

## Технические детали

- Поддерживаемые языки: Python, Rust, TypeScript, JavaScript, Go
- Размер файлов: < 1 MB для индексации
- Исключения: `.git`, `node_modules`, `venv`, `__pycache__`, `.zed`
- Эмбеддинги: LM Studio / Ollama (fallback)
- **Multi-Window (v2.3+):** Несколько открытых проектов в Zed получают
  изолированные Indexer-ы через `ProjectIndexerRegistry` (LRU 5).
  ResourceMonitor адаптивно снижает скорость индексации при высокой
  RAM/CPU нагрузке. `get_health_report` показывает registry stats
  (cached/evictions/hits/misses) и process_rss_mb/cpu_percent.
