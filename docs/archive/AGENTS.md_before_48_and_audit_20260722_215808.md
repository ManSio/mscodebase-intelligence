# УНИВЕРСАЛЬНЫЙ ПРОТОКОЛ АРХИТЕКТУРЫ, МЫШЛЕНИЯ И САМОКОРРЕКЦИИ АГЕНТА

## 0.-1. ПРИОРИТЕТ ПРАВИЛ И НАСЛЕДОВАНИЕ SKILLS

Этот файл (`AGENTS.md`) — верхний уровень. Любой Skill-файл (`architecture-review`,
`bug-hunting`, `code-search-analysis`, `safe-refactoring`, `test-creation`, `systematic-cross-check`
`documentation-update`, и любой будущий) — это **workflow**, не замена протокола.

**Skill определяет ПОСЛЕДОВАТЕЛЬНОСТЬ шагов. `AGENTS.md` определяет, когда шаг
считается пройденным и что обязано быть в финальном отчёте.**

Если Skill-файл содержит свой формат `[🏁 ИТОГ ДЛЯ ТЕБЯ]` — этот формат
**дополняется**, а не заменяется, следующими полями из §8:
- `Гипотеза / Эксперимент` (§1 — если задача involved выбор между подходами)
- `Concurrency / race notes` (§5.13, если применимо)
- `Числа` (§5.15 — замер+команда, или "не измерено")
- `Definition of Done` (§7)
- `Verified from clean state` (§7.7)

Любой Skill, чей отчёт не содержит эти поля (там где применимо) — считается
нарушившим протокол, даже если сам Skill формально выполнен по своим шагам.
В конце Workflow каждого Skill-файла должна быть строка: "Итоговый блок
наследует §8 AGENTS.md — обязательные поля выше применяются вне зависимости
от формата, описанного в этом Skill."

## 0.0. MCP-FIRST PROTOCOL (MSCodeBase Intelligence)

Если в среде доступны MCP-инструменты MSCodeBase Intelligence
(`search_code`, `get_symbol_info`, `impact_analysis`, `intel_get_runtime_status`
и др.), агент ОБЯЗАН работать через них, **а не через grep/read_file**.

**Порядок проверки:**
1. Вызвать `debug_runtime_passport()`. Ответ с RUN_ID/BUILD_ID → MCP жив.
2. Если жив — MCP-инструменты как PRIMARY поиск.
3. Если transport error / "server shut down" → fallback на grep + read_file + terminal.

**Замена IDE-инструментов на MCP:**

| Задача | ❌ Не использовать | ✅ MCP |
|--------|---|---|
| Найти код по смыслу | grep | `search_code(mode="quality")` |
| Точное имя символа | grep | `search_code(mode="fast")` / `get_symbol_info` |
| Архитектура/связи | читать файлы | `intel_code_topology` / `impact_analysis` |
| Символ: кто вызывает | grep | `get_symbol_info` |
| Прочитать файл | read_file | `read_live_file` |
| Состояние проекта | shell/cat | `intel_explain_project_state` |
| Логи/ошибки | cat | `get_logs` / `intel_predict_root_cause` |
| Git история | git log | `git(action="log")` / `git(action="history")` |

После вызова критических фоновых операций (в частности, intel_trigger_reindex()), агент обязан незамедлительно выполнить python scripts/monitor.py для отслеживания прогресса в реальном времени.

Если MCP жив — **запрещено** использовать grep/read_file для поиска кода.
Fallback на IDE-инструменты разрешён только при transport error/таймауте.


⚠️ **MULTI-WINDOW CHECK:** `intel_get_runtime_status().project_path`
показывает проект, в котором MCP был запущен. Если пользователь говорит
о ДРУГОМ проекте (открыт в другом окне) — **НЕ ДОВЕРЯЙ** данным
`search_code()`/`get_symbol_info()` для этого проекта. Вместо этого:
- Проверь, открыт ли проект: `ls <путь>`.
- Предупреди пользователя, предложи переключиться.
- Используй `intel_explain_project_state` для быстрой диагностики.

**ТРАНСПОРТНЫЙ ПРОТОКОЛ:** вызовы инструментов — строго JSON, без XML-тегов:
```json
{"name": "имя_инструмента", "arguments": {"параметр1": "значение1"}}
```

**Строгая последовательность вызовов:** никогда не отправлять несколько
MCP-инструментов параллельно в одном блоке. Дожидаться полного ответа одного
инструмента перед вызовом следующего.

**JSON Payload Hygiene:** запрещены рекурсивные glob-паттерны (`**`, `**/*.py`)
в `include_pattern`. Только прямой путь или плоский список файлов.

**Parameter Minimization:** не указывать необязательные параметры
(`timeout_ms` и т.п.), если они не критичны. При ошибке парсинга — сначала
убрать все второстепенные параметры, повторить в минимальной комплектации.

**Self-Correction при ошибке JSON:** не повторять ту же команду. Признать
ошибку → упростить (убрать glob, убрать необязательные параметры) → разбить
сложную команду на последовательные атомарные вызовы.

**Git Tools:** единый инструмент `git` с параметром `action`:
`git(action="log", limit=5)`, `git(action="history", file_path=...)`,
`git(action="branch")`.

## 0.1. TASK STATE PERSISTENCE (Context Recovery)

**Проблема:** при сжатии контекста агент теряет историю задачи и начинает
всё заново — переделывает готовое или ломает.

**Решение:** `.agent_task_state.md` в корне проекта.

**Протокол:**

1. **Создание (`[📝 ПЛАН]`):** записать `.agent_task_state.md`:
   - `# Task` — описание
   - `## Hypothesis` — если задача исследовательская: что проверяем и как поймём результат
   - `## Plan` — шаги `[done]`/`[active]`/`[pending]`
   - `## Files Changed` — что и зачем
   - `## Key Decisions` — почему выбрано то или иное
   - `## Next Action` — что делать прямо сейчас
   - `## Danger Zone` — что НЕ трогать

2. **Обновление:** перед/после `[⚙️ КОДИРОВАНИЕ]` — статусы шагов; после
   `[✅ ПРОВЕРКА]` — результат (тест OK/fail, гипотеза подтверждена/опровергнута).

3. **Восстановление (после сжатия контекста):**
   1. Прочитать `.agent_task_state.md` (если есть).
   2. Сверить с `AGENT_DIARY.md` и состоянием файлов.
   3. Продолжить с `## Next Action`.
   4. Если файла нет — задача завершена, начать новую по команде.

4. **Удаление (`[🏁 ИТОГ]`):** удалить файл, запись в `AGENT_DIARY.md`.
   Если задача приостановлена — оставить `[paused]` в имени.

**Handoff:** `.agent_task_state.md` — в `.gitignore` (проверить при создании,
если нет строки — дописать).

### 0.1.1. VERIFICATION LEDGER (Реестр верификации и исследований)

**Проблема ("Lost in the Middle"):** при разборе длинного списка задач (аудит
багов, рефакторинг по пунктовому списку, инкрементальная проверка гипотез)
агент теряет статус "подтверждено/опровергнуто", если сжатие контекста
происходит до завершения всех пунктов. В длинных сессиях (10+ пунктов)
деградация точности достигает 40-60% (Anthropic, SWE-agent benchmarks).

**Решение:** для задач, где проверяется **≥3 независимых гипотез или пунктов**,
агент ОБЯЗАН завести таблицу `## Verification Results` внутри
`.agent_task_state.md`.

**Протокол обновления:**
1. Агент обновляет эту таблицу **после** проверки каждого пункта, а не в конце.
2. Статусы: `🔍 IN PROGRESS`, `✅ CONFIRMED`, `❌ REFUTED`, `⏳ DEFERRED`, `⛔ BLOCKED`.
3. Если контекст был сжат, первое действие агента — прочитать эту таблицу и
   продолжить с первого статуса `🔍 IN PROGRESS` или следующего `pending`.
4. Каждая строка **обязана** содержать `File:Line` — конкретное место, где
   проверялся пункт. Гипотеза без локализации = не проверенная гипотеза.

**Обязательный формат таблицы в `.agent_task_state.md`:**
```markdown
## Verification Results
| # | Issue / Hypothesis | File:Line | Status | Verdict / Evidence |
|---|--------------------|-----------|--------|--------------------|
| 1 | Deadlock in add() | rate_lim.py:120 | ✅ CONFIRMED | Wait inside lock — repro test added |
| 2 | wmic deprecated | layer.py:226 | ❌ REFUTED | Already uses `platform.processor()` |
| 3 | Race in pool submit | pool.py:88 | 🔍 IN PROGRESS | — |
| 4 | OOM on large index | indexer.py:340 | ⏳ DEFERRED | Needs 50GB+ dataset |
```

**Правило фокуса:** Если агент начал исследовать новую гипотезу, не
отмеченную в таблице, он обязан сначала добавить её со статусом
`🔍 IN PROGRESS`. Это предотвращает "прыжки" между задачами и
незакрытые циклы (§9.10).

**Связь с §7 (Definition of Done):** Для задач с Verification Ledger,
статус `✅` в `[🏁 ИТОГ]` разрешён только если все строки таблицы
имеют статус `✅ CONFIRMED` или `❌ REFUTED` (оба = "закрыто").
Наличие `🔍 IN PROGRESS` или `⏳ DEFERRED` без явного обоснования
понижает статус до `⏳ частично`.

**Восстановление после перерыва (preemption/resume):**
1. Прочитать `.agent_task_state.md` → секцию `## Verification Results`.
2. Найти первую строку со статусом `🔍 IN PROGRESS` или первый `pending`.
3. Продолжить с этого пункта — не начинать заново, не пропускать.
4. Если все строки закрыты — переходить к `## Next Action` / `[🏁 ИТОГ]`.

## 0.2. ИСПОЛНЯЕМАЯ, А НЕ ДЕКЛАРАТИВНАЯ ПРОВЕРКА

`verified_from_clean_state` — это результат **реального запуска**
`scripts/verify_clean_state.sh`, не текстовое поле по памяти.

1. Если есть terminal/bash с сетью — агент ОБЯЗАН реально вызвать скрипт
   перед "✅" и вставить в отчёт **сырой хвост вывода**, не пересказ.
2. Если terminal недоступен / нет сети / transport-ошибка — агент НЕ
   имитирует проверку. Статус понижается до `⚠️ не проверено — <причина>`,
   владельцу явно предлагается прогнать скрипт вручную.
3. Разница между "не проверено, потому что не мог" и "не проверено, потому
   что не проверил" — обязательна в тексте. Первое — честная граница
   возможностей инструмента. Второе — нарушение протокола.

```bash
#!/usr/bin/env bash
# scripts/verify_clean_state.sh — единственный источник правды "проект реально работает"
set -uo pipefail
TMPDIR=$(mktemp -d)
trap 'rm -rf "$TMPDIR"' EXIT

REPO_URL="${1:?Usage: verify_clean_state.sh <repo_url>}"

git clone --depth 1 "$REPO_URL" "$TMPDIR/repo" -q || { echo "CLONE_FAILED"; exit 2; }
cd "$TMPDIR/repo"
python3 -m venv venv -q
venv/bin/pip install -q -e . 2>&1 | tail -20
RESULT=$(venv/bin/python -m pytest tests/ -q --timeout=60 2>&1)
EXIT_CODE=$?
echo "$RESULT" | tail -15
echo "CLEAN_STATE_EXIT_CODE=$EXIT_CODE"
exit $EXIT_CODE
```

Важно: команда `pytest tests/` — **без указания конкретных файлов**. Выборочный
прогон (`pytest tests/test_a.py`) не ловит `ImportError`/`IndentationError` в
несвязанном файле, который блокирует всю коллекцию тестов (см. §9, п.2).

## 1. ПРОТОКОЛ ОБОСНОВАННОГО ВЫБОРА И ЭКСПЕРИМЕНТОВ (Research → Experiment → Decision)

**Главное правило:** владелец принимает финальные архитектурные решения, но
агент обязан подготовить **фактологическую базу**. Никаких теоретических догадок,
особенно в задачах формата "что быстрее/лучше/надёжнее" — эти задачи существуют
именно для того, чтобы их проверяли экспериментом, а не интуицией.

### Шаг 1: [🔍 ИССЛЕДОВАНИЕ] (Research)(поиск через интернет)
- Найти, как задачу решают в топовых open-source проектах (best practices,исследования, эксперементы, изучить код других проектов).
- Найти известные подводные камни (edge cases, производительность, утечки памяти).
- Сформулировать **явную, проверяемую гипотезу** до эксперимента: "Ожидаю X ch/s
  при N потоках, потому что Y" — не просто "проверим, что получится".

### Шаг 2: [🧪 ЭКСПЕРИМЕНТ] (Experimentation)
- **Не предлагать теорию, предлагать замер.** Минимальный benchmark/тест прямо в проекте.
- Замерить реальные метрики: скорость (ms), RAM/CPU, точность.
- Результаты — в `EXPERIMENTS_LOG.md` (гипотеза → команда запуска → сырой вывод → вывод).
- **Правило одного бенча:** один изолированный замер НЕ является доказательством
  для продакшн-поведения. Если гипотеза объясняет расхождение между "тестовым
  стендом" и "реальным рантаймом" ("это артефакт fresh-compile, в реальности
  не повторится") — такая гипотеза обязана быть подтверждена ОТДЕЛЬНЫМ
  контролируемым тестом именно этого расхождения (прогон через реальный
  вызываемый путь, не через изолированный вызов внутреннего API), а не принята
  как объяснение по умолчанию. Без такого теста формулировка в `AGENT_DIARY.md`
  обязана быть "предположение, не проверено", а не "причина".
- **Правило контрольной группы:** если эксперимент сравнивает "было" и "стало"
  (например: было — единый лок, стало — пул) — оба варианта измеряются той же
  методикой, тем же числом потоков, в той же сессии. Нельзя сравнивать старое
  число из прошлой недели с новым числом сегодня, если методика замера могла
  измениться.
- **Правило контроля правильности результата, не только скорости.** Если
  эксперимент касается конкурентного доступа к общему состоянию — метрика
  "ch/s" или "0 errors" недостаточна. Обязательна отдельная проверка: правильный
  вход попал в правильный выход (см. §5.13, §9 п.5).

### Шаг 3: [📊 ПРЕДЛОЖЕНИЕ ВЫБОРА] (Data-Driven Options)
Максимум 2-3 варианта, строго в формате:
1. **Вариант А:** (Название)
   - *Как работает:* (1 предложение)
   - *Плюсы:* (на основе эксперимента, с цифрой и командой замера)
   - *Минусы/Риски:* (на основе исследования)
   - *Рекомендация агента:* (чёткий вердикт)
2. **Вариант Б:** ...

### Шаг 4: [✅ РЕШЕНИЕ ВЛАДЕЛЬЦА]
Агент ждёт простого ответа ("Делаем А"). После команды → `[⚙️ КОДИРОВАНИЕ]`.

### 1.5. Запрет на архитектурную метрику без сквозного замера
Любое число вида "throughput ×N" или "X ch/s" обязано быть получено прогоном
**полного пути вызова** (например `embed_batch()` целиком, а не `.infer()`
напрямую) — не описанием архитектуры ("N слотов в пуле = N× параллелизм").
Архитектурное описание — гипотеза, не результат. Запись в `EXPERIMENTS_LOG.md`
обязана содержать: команду запуска, число потоков, реальные ch/s, явную графу
"Errors" с расшифровкой, что именно посчитано как ошибка.

### 1.6. Реестр гипотез (Hypothesis Ledger)
Каждая проверенная гипотеза попадает в `EXPERIMENTS_LOG.md` в формате:

```markdown
## [YYYY-MM-DD] — Гипотеза: <короткая формулировка>
**Ожидание:** <что должно получиться и почему>
**Команда:** <точная команда запуска>
**Сырой результат:**
```
<вывод команды, не пересказ>
```
**Вердикт:** подтверждена / опровергнута / частично (указать что именно)
**Если опровергнута прошлая гипотеза** — см. §4 п.5 (запрет на реверт без опровержения).
```

Гипотеза без сырого вывода команды — не гипотеза, а предположение, и обязана
быть помечена как таковое в тексте (см. §5.15).

## 2. АРХИТЕКТУРА "ТУМБЛЕР" (The Tumbler Pattern)

### 2.1. Строгое разделение ответственности
1. **Конфигурация:** все переменные, пути, ключи — строго в `.env`, читаются
   через `config.py`. Никаких hardcoded values в коде.
2. **Ядро (Core):** знает только ЧТО нужно сделать. Абстрактные интерфейсы.
3. **Провайдеры (Providers/Adapters):** конкретная реализация. Смена
   технологии = одна строка в конфиге.

### 2.2. Универсальная структура проекта
```text
project_root/
├── .env.example
├── config.py
├── core/
├── interfaces/
├── providers/
├── utils/
├── tests/
├── docs/
├── AGENT_DIARY.md            # единственный дневник — не DEV_DIARY.md
├── EXPERIMENTS_LOG.md        # гипотеза → замер → вывод (см. §1.6)
└── KNOWN_ISSUES.md           # синхронизируется из AGENT_DIARY при каждом [🏁 ИТОГ], см. §4.6
```

### 2.3. Правило общего мутабельного состояния (Shared Mutable State)
Если провайдер вводит общий объект, к которому обращаются из нескольких
вызовов конкурентно (pool, queue, cache, shared dict, InferRequest/
AsyncInferQueue и т.п.) — ОБЯЗАТЕЛЬНО:
1. Корреляционные ключи (`userdata`, `request_id`, task-id) — уникальны
   **на вызов**, не переиспользуемые локальные индексы между вызывающими потоками.
2. Результаты конкурентных операций собираются в **локальный для вызова**
   объект, а не в общий атрибут инстанса (`self._results`), если только
   доступ к нему не защищён явным механизмом изоляции по вызову.
3. Замена одного thread-safety механизма на другой (lock → pool → queue) НЕ
   считается закрытой без теста на **корректность содержимого** результата
   под конкуренцией, а не только на отсутствие исключений. "0 errors" ≠
   "данные верные" — тест обязан сверять, что вернувшийся результат
   принадлежит правильному входу.

### 2.4. Бюджет размера файла (God Object Guard)
Файл в `core/`, `providers/` свыше **800 строк** — триггер на декомпозицию в
следующей сессии редактирования (не обязательно немедленно, но нельзя
добавлять новый функционал без пометки в `.agent_task_state.md`/`AGENT_DIARY.md`
о причине отсрочки). Решение НЕ декомпозировать — явно фиксируется как
**осознанный техдолг** с обоснованием в `KNOWN_ISSUES.md`.

## 3. ВИДИМЫЙ ПРОЦЕСС МЫШЛЕНИЯ (Agentic Loop & Self-Correction)

**💾 Восстановление контекста (первый вызов после сжатия):** проверить
`.agent_task_state.md`. Если есть — сверить с `AGENT_DIARY.md` и состоянием
файлов, продолжить с `## Next Action`.

- **[🔍 ИССЛЕДОВАНИЕ]**: читаю `AGENT_DIARY.md`, `EXPERIMENTS_LOG.md`. Ищу похожие решения.
- **[🧠 АНАЛИЗ]**: подводные камни, соответствие "Тумблеру", достаточность контекста.
- **[📝 ПЛАН]**: кратко (3-4 пункта). Обновляю `.agent_task_state.md`, включая `## Hypothesis`, если применимо.
- **[⚙️ КОДИРОВАНИЕ]**: минимальный diff. Никаких TODO/stubs. Production-ready код.
- **[🧪 ВЕРИФИКАЦИЯ]**: `pytest`/benchmark, `diagnostics`. Если тестов нет — пишу минимальный воспроизводящий тест.
- **[✅ ПРОВЕРКА]**: тесты/бенчмарки в реальной среде (см. §7 — без этого шага статус не "✅").
- **[🔄 САМОКОРРЕКЦИЯ]**: тест упал — НЕ повторяю то же действие. Читаю логи, меняю гипотезу. Лимит: после 2 разных гипотез без успеха — СТОП, спросить владельца.
- **[📚 ДОКУМЕНТАЦИЯ]**: обновляю (см. §6).
- **[🏁 ИТОГ]**: финальный блок (см. §8).

**Параллельность вызовов:** независимые действия — за один шаг. Нельзя
параллелить, если результат одного нужен для аргументов другого.

### 3.1. Запрет на "✅ Статус" без проверки с чистого состояния
Перед `[🏁 ИТОГ]` со статусом ✅ для фикса, затрагивающего конкурентность,
смену модели/зависимости, перенос файла между пакетами, изменение маппинга
install/config — агент ОБЯЗАН выполнить и указать: `verified_from_clean_state:
<команда, которой перепроверено>`. Без этой строки статус — `⚠️ требует
внешней проверки`, не `✅`.

## 4. ПРОТОКОЛ ИССЛЕДОВАНИЯ И РАЗБОРА ПОЛЁТОВ (Post-Mortem)

1. **Локальный аудит:** перед кодом — проверить `AGENT_DIARY.md`. Не решалась ли задача раньше?
2. **Внутренний поиск:** существующие хелперы, которые можно расширить.
3. **Фиксация эксперимента** в `EXPERIMENTS_LOG.md` (см. §1.6).
4. **Post-Mortem** в `AGENT_DIARY.md`: *Симптом → Root Cause → Fix → Guard.*
5. **Запрет на реверт без опровержения.** Если сессия отменяет вывод
   предыдущей ("предыдущий агент ошибся, на самом деле...") — запись обязана
   явно указывать, **каким тестом опровергнут** предыдущий вывод. Без теста —
   реверт запрещён, фиксировать как "открытое противоречие", выносить на
   решение владельца.
6. **`AGENT_DIARY.md` ↔ `KNOWN_ISSUES.md` синхронизация.** Каждая запись
   "Root Cause + Fix" — в тот же коммит получает зеркальную строку в
   `KNOWN_ISSUES.md` (дата, что было, статус). `KNOWN_ISSUES.md`, не
   обновлявшийся дольше одной сессии — нарушение протокола.
7. **Единственность дневника.** В проекте существует ровно один файл-дневник
   — `AGENT_DIARY.md`. Обнаружение второго файла аналогичного назначения
   (`DEV_DIARY.md`, `NOTES.md`, любой другой) — само по себе повод для
   Post-Mortem: смёрджить в канонический, оставить архивный файл как явный
   редирект с текстом "ARCHIVED — см. AGENT_DIARY.md", не удалять молча.

### 4.8. КОНТРОЛЬ РАЗДУВАНИЯ ДНЕВНИКА (BLOAT CONTROL) И АРХИВАЦИЯ

**Проблема:** Дневник (`AGENT_DIARY.md`) имеет свойство превращаться в дамп сырых логов и огромных кусков кода. Это убивает его эффективность: при загрузке в контекст он вытесняет полезные инструкции, а агент перестаёт находить нужные факты (эффект "Lost in the Middle").

**Правило 1: Лимит плотности (15 строк).**
Одна запись в `AGENT_DIARY.md` не должна превышать **15 строк** (включая заголовок и пустые строки). Если инцидент сложный — ссылайся на `EXPERIMENTS_LOG.md` или commit, но не вставляй всё в дневник.

**Правило 2: Запрет на сырые дампы (No Raw Dumps).**
Строго запрещено вставлять в дневник:
- Выводы `pytest` (кроме одной строки `519 passed, 0 failed`).
- Stack trace'ы длиннее 3 строк.
- Куски кода длиннее 5 строк (код уже есть в репозитории, в дневнике нужен только `file.py:L120`).
- Сырые логи терминала. Пиши *суть* лога, а не копируй его.

**Правило 3: Формат "Вердикт-Сначала" (Conclusion-First).**
Каждая запись обязана следовать жёсткому шаблону (без лирики и процесса расследования):
```markdown
## [YYYY-MM-DD HH:MM] — КРАТКОЕ ИМЯ ИНЦИДЕНТА
**Status:** ✅ Fixed / 🟡 Partial / 🔴 Open
**Root Cause:** (1-2 предложения, почему сломалось)
**Fix:** (1-2 предложения, что сделали + commit hash)
**Guard:** (Как предотвратить в будущем — правило, тест или file:line)
```

**Правило 4: Архивация (Monthly Rotation).**
Раз в месяц (или если файл превышает 300 строк) агент ОБЯЗАН:
1. Перенести все записи старше 1 месяца в `docs/archive/AGENT_DIARY_YYYY_MM.md`.
2. Оставить в `AGENT_DIARY.md` только записи текущего месяца.
3. (Опционально) Добавить в начало файла раздел `## Key Historical Decisions` с 5-10 буллитами важнейших архитектурных сдвигов (например: "Перешли с asyncio.Lock на threading.Lock для shared state").

**Правило 5: Индекс, а не хранилище.**
`AGENT_DIARY.md` — это оглавление инцидентов и ADR (Architectural Decision Records). Сырые данные, графики замеров и пошаговые логи расследований живут в `EXPERIMENTS_LOG.md`.

## 5. ЖЁСТКИЕ ПРАВИЛА РАБОТЫ С КОДОМ И БЕЗОПАСНОСТЬ

1. **Минимальный diff.** Не переписывать файл целиком ради 3 строк.
2. **Запрет на галлюцинацию.** Не угадывать содержимое/номера строк. Сначала
   MSCodeBase Intelligence (если доступны), иначе `read_file`/`grep`.
3. **Правило одной попытки.** Ошибка редактирования — не повторять ту же команду.
4. **Наследование защиты.** Новый код — те же guards/fallbacks/конфигурация, что и рядом.
5. **Безопасность.** Никогда `rm -rf`, `drop database`, `push --force` без явной команды.
6. **Секреты.** Только в `.env` через `config.py`. Найден секрет в коде — не коммитить, предупредить.
7. **Push — только по команде.** `git commit` без спроса. `git push` — только по явному "запушь"/"пуши"/"push".
8. **Откат:**
   a) Незакоммиченное: `git checkout -- <файлы>`.
   b) Закоммиченное, не запушенное: `git reset --soft HEAD~1`.
   c) Запушенное: сообщить владельцу — ручной `git revert`.
9. **ENCODING SAFETY (Windows):** первые строки любого `execute_script`:
   ```python
   import sys
   if sys.stdout.encoding != 'utf-8':
       sys.stdout.reconfigure(encoding='utf-8')
   ```
10. **PATHLIB ONLY.** Только `pathlib.Path`, кроме аргументов MCP-инструментов (Windows-формат).
11. **TRY/EXCEPT в каждом скрипте:**
    ```python
    try:
        ...
    except Exception:
        import traceback
        traceback.print_exc()
        sys.exit(1)
    ```
12. **Path Safety при записи файлов из динамического пути:**
    ```python
    resolved = (base_path / relative_path).resolve()
    if not resolved.is_relative_to(base_path.resolve()):
        raise ValueError("path traversal detected")
    ```
    Без проверки код небезопасен вне зависимости от того, вызывается ли он сейчас откуда-либо.

13. **Протокол Concurrency / Shared State.** Если код вводит/трогает объект,
    к которому обращаются из более чем одного потока/корутины (общий клиент,
    соединение, кэш, `InferRequest`, connection pool, любой `self._x`,
    живущий дольше одного вызова):
    - Обязателен явный ответ на "что произойдёт при двух одновременных
      вызовах?" — в `.agent_task_state.md` как `## Concurrency Note`.
    - Обязателен конкурентный стресс-тест (N ≥ 2 и N ≥ 8), проверяющий не
      только отсутствие исключений, но **корректность результата** —
      правильный вход попал в правильный выход, а не просто "не упало".
    - Замена одного примитива на другой (объект+lock → пул+callback) не
      закрывает пункт автоматически — пул/async API создаёт собственную
      поверхность для гонки (общий словарь результатов, общие id
      корреляции), проверяется тем же стресс-тестом заново.

14. **Протокол миграций и переименований.** Перенос файла, переименование
    класса/функции, смена сигнатуры публичного API — не закончены, пока не
    выполнено ВСЁ:
    - `grep -rn` по старому имени/пути по всему репозиторию (src, tests,
      scripts, docs, .env.example, **включая сам AGENTS.md и Skill-файлы**).
    - Каждое найденное вхождение — обновлено или явно обосновано (backward-compat shim).
    - Любой shim — с дедлайном удаления в `KNOWN_ISSUES.md` (дата/версия, не "потом").
    - Комментарии/докстринги, ссылающиеся на удалённые функции — запрещены.

15. **Протокол числовых утверждений.** Любое числовое утверждение о
    производительности ("throughput ×4", "250 ch/s", "−26% строк") —
    с командой замера и сырым выводом. Число без замера — маркировано
    "ожидается, не измерено".

16. **Запрет на прямое управление MCP-процессами.** Никогда `taskkill`/`kill`/
    `os.kill` для MCP-сервера. Единственный путь перезапуска — `Ctrl+Shift+P` →
    Reload Window. Исключение: внешние дочерние процессы (`llama-server.exe`).

17. **Запрет на прямой доступ к БД извне MCP.** Никогда `lancedb.connect()`/
    `DBManager` вне контекста MCP-процесса. Все операции — только через
    MCP-инструменты. Исключение: одноразовые миграции схемы при остановленном
    MCP и явном подтверждении владельца.

18. **Мониторинг через скрипт, не через poll.** Длительные операции (>30 сек)
    — `scripts/monitor.py` (tail -f лога), не частые poll-запросы статуса.

19. **Пиновка зависимостей — с проверкой API, не по памяти.** Перед любым
    `package==X.Y.Z` в `requirements.txt`/`pyproject.toml` — прогнать
    `python -c "import pkg; print([a for a in dir(pkg) if ...])"` именно для
    версии X.Y.Z, сверить с тем, что реально использует код (не с нижней
    границей диапазона, не "должно быть похоже"). Пин без rationale-комментария
    запрещён:
    ```toml
    # Pinned: <библиотека> меняла публичный API между минорными версиями
    # (сломала <что именно> в <дата>). Бамп — только после проверки API.
    package==X.Y.Z
    ```

**§5.20 — Закрывай цикл (Close the Loop).** После любого рефакторинга/переименования/переноса файла — все импорты по всему репозиторию (src, tests, scripts, docs, AGENTS.md, Skill-файлы) обновляются в **том же коммите**. Shim-файлы (редиректы для обратной совместимости) — только если в том же коммите стоит дедлайн удаления. Без дедлайна — shim запрещён.

**§5.21 — Версионная дисциплина.** `bump_version.py --check` (pyproject.toml = CHANGELOG = все три языка) — обязателен перед любым commit, затрагивающим версию. Pre-commit hook блокирует коммит при расхождении.

**§5.22 — Верхняя граница зависимости проверяется перед коммитом, не только нижняя.** Перед добавлением/изменением `<X` к любой зависимости — выполнить `pip index versions <package>` (Linux/macOS) или эквивалент, чтобы убедиться, что текущий установленный диапазон попадает внутрь нового потолка. Массовое добавление одного потолка (`<1`) сразу ко всем пакетам в списке без проверки индивидуальной текущей мажорной версии — запрещено. Каждый пакет проверяется отдельно.

## 6. ДОКУМЕНТАЦИЯ КОТОРАЯ ЖИВЁТ (Living Documentation)

### 6.1. Структура и правила
- **README.md:** точка входа, обновляется при изменении установки/конфига.
- **CHANGELOG.md:** из commit messages (Conventional Commits).
- **ADR:** `docs/adr/`. Контекст → Решение → Почему (альтернативы) → Последствия.
- **Docstrings:** каждая публичная функция — описание, параметры, пример.
- **Tests as documentation.**
- **EXPERIMENTS_LOG.md:** гипотеза → замер → вывод (см. §1.6), при каждом эксперименте.
- **AGENT_DIARY.md:** единственный дневник инцидентов и решений (см. §4 п.7).

### 6.2. Запреты
- ❌ Документация "потом" — только в том же коммите.
- ❌ Дублирование информации (Single Source of Truth) — включая **дублирование
  самих файлов документации** (два `KNOWN_ISSUES.md` в разных папках, два
  дневника с разным содержимым). Перед созданием нового doc-файла — `find . -iname "<то же имя>"` по всему репозиторию.
- ❌ Устаревшая документация — лучше удалить (с явным редиректом), чем врать.
- ❌ Числа/версии, продублированные в 2+ файлах вручную. Версия — из ОДНОГО
  источника скриптом (`bump_version.py`), остальные файлы генерируются/сверяются.
- ❌ Дефолтное значение конфигурации, продублированное в установщике
  (`install.py`) и в рантайме (`config.py`) без единого источника или CI-теста на совпадение.
- ❌ Commit message, описывающий архитектурное изменение, которое не
  реализовано в diff'е этого коммита ("git-based locking" в сообщении, когда
  реализация — та же markdown-таблица). Сообщение коммита обязано точно
  отражать содержимое diff'а, не намерение или план.

### 6.3. Единый источник версии
Версия проекта — в ОДНОМ месте (`pyproject.toml`), остальные упоминания
(`CHANGELOG.md`, README badges) генерируются/проверяются `bump_version.py`.
Ручное редактирование версии в более чем одном файле — запрещено.

### 6.4. Ledger-проверка дневника
`AGENT_DIARY.md` не является источником правды сам по себе — каждое
утверждение "✅ реализовано X" обязано быть проверяемым автоматически
(наличие функции/класса, прохождение теста, факт коммита). Раз в сессию
(или по команде владельца) агент прогоняет такую проверку по всем
"✅"-записям за последние N дней и явно репортует расхождения.

### 6.5. Систематический кросс-чек (Systematic Cross-Check)
Раз в месяц (или по явной команде владельца) — полный проход:
1. Версия едина (`bump_version.py --check`).
2. Tool counts в README/docs vs реальная регистрация в `server_tools.py`.
3. Единственность `KNOWN_ISSUES.md` — `find . -iname "KNOWN_ISSUES.md"` не должен дать >1.
4. Все импорты из `AGENTS.md` и Skill-файлов ведут на существующие файлы (grep по путям из §6.5 предыдущих версий, устаревшие записи удалены).
5. Нет shim-файлов без дедлайна удаления.

## 7. DEFINITION OF DONE — когда статус реально может быть "✅"

Статус "✅" в `[🏁 ИТОГ]` разрешён только если пройдены ВСЕ пункты:

1. **Чистая проверка.** Изменения проверены через `git stash`/чистый временный
   клон/перезапуск процесса с нуля — минимум один раз перед финальным "✅".
   Если невозможно за один цикл — статус "✅ (локально), требует чистой проверки".
2. **Тест реального пути, не мока.** Runtime-поведение (индексация, инференс,
   сетевой вызов) — verification через реальный путь выполнения хотя бы раз.
3. **Concurrency note закрыта** (если применимо, §5.13).
4. **Grep-развёртка после структурных изменений** (§5.14) — ноль оставшихся
   ссылок на старое имя/путь, кроме явно обоснованных.
5. **Числа подтверждены** (§5.15) — команда+вывод, либо "неизмеренное ожидание".
6. **`KNOWN_ISSUES.md` синхронизирован** с этой сессией (§4.6).
7. **`verified_from_clean_state`** — строка `Verified from clean state:
   <да/нет + как именно>`. Отсутствие строки понижает статус с ✅ до ⚠️.
8. **Гипотеза (если была) закрыта явным вердиктом** — подтверждена/опровергнута
   с сырым выводом эксперимента (§1.6), не оставлена как "предположительно работает".
9. **Verification Ledger закрыт** (если применимо, §0.1.1) — все строки таблицы
   имеют статус `✅ CONFIRMED` или `❌ REFUTED`. Наличие `🔍 IN PROGRESS`
   без обоснования понижает статус до `⏳ частично`.

Если хотя бы один пункт не выполнен — статус ТОЛЬКО "⏳ частично" или
"✅ с оговоркой: <что не проверено>", никогда молчаливое "✅".

### 7.1. P0 не копить
Если в ходе сессии найден P0 — он фиксится до окончания сессии, а не
откладывается. P1 — заводится запись в `.agent_task_state.md` с планом
закрытия в той же сессии или явным решением владельца об отсрочке.

синхронизировать `KNOWN_ISSUES.md` в конце работы.

## 9. РЕЕСТР ИЗВЕСТНЫХ ЛОВУШЕК (живой список, обновлять по факту)

1. **Пиновка зависимости без проверки API диапазона.** Нижняя граница
   диапазона может не иметь атрибутов, которые использует текущий код
   (пример: `lancedb==0.12.0` не имеет `Table`, только `AsyncTable`). Перед
   пином — проверить именно ту версию, которую пиньте (см. §5.19).
2. **Полная коллекция тестов ≠ выборочный прогон.** `pytest tests/test_a.py`
   не ловит `ImportError`/`IndentationError` в несвязанном `test_b.py`,
   который блокирует `pytest tests/` целиком. "Чистая проверка" по умолчанию
   — без фильтров файлов.
3. **Codemod/перенос модуля — grep по старому имени обязателен**, включая
   комментарии, docstrings, и **сам `AGENTS.md`/Skill-файлы**, если они
   ссылаются на переименованный путь.
4. **Тест, полагающийся на реальное окружение (LSP bridge, CWD, env vars)
   вместо мока — падает по-разному в CI и на dev-машине.** Мокать все внешние
   источники явно, не полагаться на "у меня резолвится правильно".
5. **Замена одного thread-safety механизма другим (лок → пул → async queue)
   создаёт свою собственную новую поверхность для гонки** (общий словарь
   результатов, общий correlation id) — каждая замена требует нового
   стресс-теста на корректность данных, не только "не упало".
6. **Два файла одного назначения (дневник, KNOWN_ISSUES, config) в разных
   местах репозитория.** Возникает при параллельной работе агентов/сессий,
   не заметивших существующий файл. Проверка: `find . -iname "<имя>"` перед
   созданием любого doc/config файла, если есть подозрение, что он может уже существовать.
7. **Commit message заявляет архитектурное изменение, которого нет в diff'е.**
   ("git-based locking" в сообщении при реализации через ту же shared-таблицу).
   Перед коммитом — перечитать message и сверить с реальным содержимым diff'а.
8. **`git push` отклонён (`non-fast-forward`), и вместо диагностики пишется
   скрипт-обёртка для "решения" проблемы.** Правильный порядок: `git fetch`
   → `git log origin/main --oneline` → понять причину расхождения → `git rebase`
   вручную, по одной команде, глядя на реальный вывод каждой. Никаких новых
   скриптов поверх git, пока не понятна причина расхождения.
9. **Амнезия Verification Ledger.** Агент пропускает незакрытые пункты таблицы
   (§0.1.1) после сжатия контекста и декларирует `✅`, хотя `🔍 IN PROGRESS`
   всё ещё есть. **Guard:** перед `[🏁 ИТОГ]` — grep по `.agent_task_state.md`
   на `IN PROGRESS` / `DEFERRED` / `BLOCKED`.

Правило пополнения: при любом Post-Mortem (§4), если корневая причина похожа
на уже описанный здесь паттерн — **сослаться на номер пункта**, а не
переоткрывать. Если причина новая — добавить пункт 10, 11... в том же коммите.

## 10. ПРОТОКОЛ КООРДИНАЦИИ МУЛЬТИАГЕНТОВ (Multi-Agent Coordination)

> Реальная git-based блокировка — не общий markdown-файл с таблицей, а
> отдельный файл-лок на ресурс, атомарность которого гарантирует git.
> Предыдущая версия называлась "git-based locking", но использовала общий
> `.agent_handoff.md`, читаемый и записываемый обоими агентами напрямую —
> это advisory-lock с гонкой TOCTOU (оба агента читают "свободно" одновременно,
> оба пишут лок, один тихо теряет данные). Ниже — версия, которая устраняет
> этот класс гонки архитектурно, а не документирует её последствия подробнее.

### 10.0. Принцип

Каждый лок — это **отдельный файл** в `.locks/<resource_id>.lock`, который
агент **коммитит** (не просто пишет на диск). Атомарность обеспечивает git:
если два агента одновременно попытаются создать/закоммитить лок на один и
тот же ресурс, второй `git push` получит конфликт (файл уже существует в
удалённой истории) — а не тихую перезапись общей таблицы. Гонка становится
видимой и отклоняемой на уровне git, а не на уровне доверия к тому, что
"агент проверил таблицу перед записью".

### 10.1. Формат файла-лока

`.locks/<путь_с_подчёркиваниями>.lock`, например для `src/core/search/engine.py`:
`.locks/src_core_search_engine_py.lock`

```json
{
  "resource": "src/core/search/engine.py",
  "agent": "Agent A",
  "acquired_at": "2026-07-19T14:00:00Z",
  "purpose": "FTS5 integration — hybrid_search_async, reindex, close",
  "estimated_duration_min": 30
}
```

### 10.2. Захват лока (Acquire)

```bash
# 1. Всегда сначала синхронизироваться
git fetch origin main
git log origin/main --oneline -3   # убедиться, что нет чужого лока, которого не видно локально

# 2. Создать файл лока
cat > .locks/src_core_search_engine_py.lock <<'EOF'
{"resource": "src/core/search/engine.py", "agent": "Agent A",
 "acquired_at": "2026-07-19T14:00:00Z", "purpose": "FTS5 integration"}
EOF

# 3. Закоммитить и НЕМЕДЛЕННО запушить — это и есть точка атомарности
git add .locks/src_core_search_engine_py.lock
git commit -m "lock: src/core/search/engine.py by Agent A (FTS5 integration)"
git push origin main
```

**Если `git push` прошёл без ошибок** — лок реально ваш, никто не мог
захватить его параллельно (git не позволил бы второй push с тем же файлом
без предварительного pull, а если бы позволил — второй агент увидел бы
конфликт при *своей* попытке закоммитить тот же путь).

**Если `git push` отклонён (`non-fast-forward` или "path already exists"
конфликт на файле лока)** — значит кто-то уже захватил этот ресурс.
Не форсить, не удалять чужой лок. Прочитать содержимое чужого
`.locks/*.lock` (после `git pull`), увидеть `agent`/`purpose`/`acquired_at` —
и либо подождать, либо работать над другим файлом.

### 10.3. Освобождение лока (Release)

```bash
git rm .locks/src_core_search_engine_py.lock
git commit -m "unlock: src/core/search/engine.py (FTS5 integration done)"
git push origin main
```

Освобождать **сразу** по завершении работы с файлом, не в конце всей сессии
— если сессия объёмная, лок должен жить ровно столько, сколько реально идёт
редактирование конкретного файла.

### 10.4. Stale-лок (агент пропал)

Если `.locks/<file>.lock` существует, а `acquired_at` старше 2 часов:
1. Проверить `git log --all --oneline -- <resource>` — были ли коммиты от
   владельца лока после `acquired_at`. Если да — агент, скорее всего, ещё
   активен (просто не снял лок вовремя), не трогать.
2. Если коммитов нет 2+ часа — можно снять лок (`git rm` + commit +push),
   но обязательно с сообщением, явно фиксирующим это как разрыв:
   ```
   unlock: src/core/search/engine.py (stale, no activity from Agent A for 2h+)
   ```
3. Никогда не снимать чужой лок молча — сообщение коммита обязано отражать причину.

### 10.5. Регистрация и обмен статусом (Level 1 — что реально нужно, без общего файла)

Вместо общей таблицы "кто чем занят" — каждый агент фиксирует свой прогресс
**коммитами** (не отдельным координационным файлом, который нужно
синхронизировать вручную). `git log --oneline --all -20` — это и есть
актуальный статус: кто что закоммитил, когда, с каким сообщением. Это
устраняет весь класс проблем из §10.11 предыдущей версии (STALE/CORRUPTED/
RECOVERED/heartbeat-таймстампы) — они существовали только потому, что статус
хранился в файле, требующем ручной синхронизации, а не в самом git-логе,
который синхронизирован по определению.

### 10.6. Архитектурные решения между агентами (Level 4, сохраняется)

Решения, которые влияют на обоих агентов — по-прежнему нужен явный
артефакт, потому что это не факт о файлах, а обсуждение. Формат — ADR
(`docs/adr/NNNN-<тема>.md`), не отдельный "handoff"-файл:

```markdown
# ADR-0012: FTS5 integration approach

## Context
Нужна полнотекстовая поддержка в hybrid_search.

## Decision
FTS5Mixin (Agent A), консистентно с существующим BM25Mixin.

## Alternatives considered
...

## Impact
Agent B не трогает engine.py, работает в layer.py (token_savings panel).

## Status
✅ implemented — commit abc1234
```

ADR коммитится и пушится как обычный файл — конфликт при одновременном
создании двух ADR с одним номером решается тем же git-механизмом (второй
push будет отклонён/смёржен, номер надо будет изменить), не требует
отдельного протокола арбитража.

### 10.7. Чеклист перед `edit_file`/`write_file`

```markdown
- [ ] git fetch origin main
- [ ] git log origin/main --oneline -3 (нет ли чужого свежего лок-коммита на мой файл?)
- [ ] Файл-лок на нужный ресурс не существует в .locks/? (если существует — не редактировать)
- [ ] Создать + закоммитить + запушить свой лок ДО начала редактирования
- [ ] После push лока — редактировать
- [ ] После завершения — снять лок (commit+push) немедленно
```
