## LEVEL 8: RECOVERY PROTOCOLS

### Scenario 1: Context Compression (Agent Lost Memory)
**Symptom:** Agent starts task from scratch, doesn't remember what it was doing.

**Recovery:**
1. Read `.agent_task_state.md` (if exists)
2. Compare with `git status --short`
3. Continue from `## Next Action`
4. If no task state file → ask owner

### Scenario 2: Handoff File Corrupted
**Symptom:** Parse error, malformed table, unexpected characters.

**Recovery:**
1. DO NOT try to fix
2. Read `git log --oneline -5` — what changed
3. Recover from git: `git show HEAD:.agent_handoff.md`
4. If git fails → create new handoff from `git status`
5. Add signal: `⚠️ RECOVERED at YYYY-MM-DDTHH:MM`

### Scenario 3: Agent Crashed (Stale Locks)
**Symptom:** Lock `infinite`, but agent hasn't updated heartbeat > 30 min.

**Recovery:**
1. Check `git status` — uncommitted changes from this agent?
2. If YES → DO NOT TOUCH (agent might return)
3. If NO → check `git log` — did agent commit?
4. If YES → safe to release locks
5. Add signal: `⚠️ STALE: Agent X last active YYYY-MM-DDTHH:MM`

### Scenario 4: Two Agents Want Same File (Conflict)
**Symptom:** Both agents wrote `REQUEST` for same file in Level 3.

**Recovery:**
1. First REQUEST (by timestamp) gets priority
2. Second agent writes `ACKNOWLEDGED` → waits
3. First agent finishes → writes `NOTIFY: RELEASED`
4. Second agent sees `RELEASED` → takes the file

**Arbitration rule:** Earlier timestamp wins. No exceptions.

### Scenario 5: Handoff Not Updated > 1 Hour (Stale State)
**Symptom:** `Last updated` > 1 hour ago.

**Recovery:**
1. Check `git status` — changes present?
2. If YES → agent worked but didn't update handoff → warn
3. If NO → agent likely finished → safe to update handoff
4. NEVER assume "done" without explicit `status: done`

### Scenario 6: Missing Handoff File
**Symptom:** `.agent_handoff.md` doesn't exist.

**Recovery:**
1. Check `git log --all --full-history -- .agent_handoff.md`
2. If existed → recover from git
3. If never existed → create new
4. Notify owner

### Scenario 7: Race Condition (Simultaneous Read)
**Problem:** Both agents read handoff at same time, both decide to lock same file.

**Recovery:** Timestamp-based arbitration:
1. Each REQUEST has timestamp
2. On conflict — earlier timestamp wins
3. Later agent gets `DENIED` or `WAITING`

**Rule:** Never `edit_file` without checking Level 2 (LOCKING).
