Отталкиваюсь не от абстрактных "AI-фич", а от того, что реально уникально именно в вашем случае: у вас единственный в своём роде rich dataset — `AGENT_DIARY.md` с историей реальных диагностических ошибок AI-агентов за 5+ дней, плюс граф кода, плюс эмбеддинги. Большинство идей ниже используют именно это сочетание, которого нет ни у одного code-intelligence проекта на GitHub — там либо только граф (Sourcegraph), либо только эмбеддинги (Cursor/Copilot indexing), но нигде нет **истории агентских диагностических ошибок как структурированных данных**.

## 1. Trust Decay — доверие как first-class атрибут символа, а не только код

Сейчас граф хранит "что вызывает что". Добавьте атрибут на каждый symbol/файл: **"когда этот код последний раз реально проверялся человеком"** (не коммитился — а именно верифицировался: тест прошёл + человек подтвердил в чате/PR review). Это git-blame наоборот: не "кто написал", а "насколько ещё свежо доверие". MCP-инструмент при выдаче ответа агенту добавляет: *"эта функция не проверялась человеком 11 дней, последний раз её трогал автономный agent-фикс"*. Это прямое лекарство от вашего собственного паттерна за эту неделю — "✅ done" без сверки с чистым клоном. Насколько я знаю, такого нет нигде: existing tools показывают git blame (кто автор), но не "уровень доверия", который затухает со временем и особенно быстро — после автономных AI-правок без ревью.

## 2. Contradiction Ledger — автоматизировать то, что я делал руками весь этот чат

Каждый раз, когда я перескачивал репо и сверял "✅ done" из диневника с реальным кодом — я находил расхождение (пул, который не запушен; функция, которой нет; модель, которую install.py не ставит). Это можно автоматизировать: MCP-инструмент парсит `AGENT_DIARY.md`, вытаскивает каждое утверждение вида "Х ✅ реализовано", генерирует из него **проверяемое условие** (grep на наличие класса/функции, факт коммита, прохождение конкретного теста) и гоняет это на каждом старте сервера. Если утверждение из дневника перестало быть правдой (как с пулом) — сервер сам кричит об этом при следующем открытии проекта, а не через два дня, когда кто-то случайно спросит "а ты уверен?". Это буквально автоматизация роли, которую в вашей паре человек-агент сейчас играете вы лично — снятие с вас функции "ловца галлюцинаций о собственном прошлом".

## 3. Shadow Canary для замены моделей — вместо ручных EXPERIMENTS_LOG.md записей

Корень провала с INT8-моделью (5 дней чинили не то) — в том, что переход на новую модель верифицировался вручную, разово, на изолированном бенче. Правильная инженерная практика, которой у вас пока нет: при смене embedder'а — не переключать сразу, а **параллельно (shadow)** прогонять и старую, и новую модель на N реальных живых запросах, сравнивать через простую метрику (cosine similarity к канареечному набору из 20-30 известных пар "запрос → правильный чанк"), и переключать дефолт только когда новая модель статистически стабильно лучше/не хуже. Это не эксперимент "было бы неплохо" — это прямая контрмера конкретному инциденту, который стоил вам пяти дней.

## 4. "Verification cost" вместо просто confidence score

Explainability-слой, который вы уже построили, отвечает "почему этот результат ранжирован так". Следующий шаг, которого я не видел ни в одном инструменте: рядом с ответом — не просто score, а **"стоимость проверки"**: "чтобы убедиться, что этот ответ верный, вам нужно посмотреть 2 файла (X, Y), это займёт ~3 минуты" — оценка не корректности, а трудозатрат на верификацию, на основе графа зависимостей и размера затронутого кода. Это прямо ложится на ваш заявленный modus operandi ("architect, QA lead, verifier") — инструмент не просто отвечает, а говорит, сколько усилий с вашей стороны нужно, чтобы не повторить историю с "три раза поверили не проверив".

## Путь к идеалу — не абстрактный, а gate-процесс из фактических провалов этой недели

Идеал здесь — не "добавить больше фич", а закрыть ровно те 6 дыр, которые реально произошли, одним механическим правилом каждая:

1. **Gate на "✅ Status"**: диневник не принимает статус "✅" без поля `verified_from_clean_clone: <commit_hash>` — то есть текстовое обязательство, что было пересобрано с нуля.
2. **Gate на смену модели**: не мержится без shadow-canary отчёта (пункт 3 выше).
3. **Gate на структурные изменения (переезд файла/пакета)**: codemod-скрипт обязателен, а не "перенесли и забыли обновить импорты" — как было с `src.core.cypher_engine` даже в сегодняшнем коде.
4. **Единая версия**: `bump_version.py`, обязателен в CI как проверка, а не ручное редактирование трёх файлов.
5. **Дедлайн на техдолг**: 17 shim-файлов и ~80 мёртвых методов, которые "не трогали (долгосрочный техдолг)" — либо конкретная дата удаления, либо они становятся вечными, как это уже происходит.

Если интересно — могу с любого из четырёх пунктов (1-4) написать конкретную реализацию: например, скрипт `verify_diary.py`, который парсит `AGENT_DIARY.md` и реально гоняет проверки против `HEAD`.

Ни один из пунктов не является переизобретением GraphRAG/Sourcegraph — это специфичные для вашей комбинации данных эксперименты.

## 1. Bug-pattern vector propagation — используйте свежий инцидент как готовый датасет прямо сейчас

Классический word2vec-трюк "king - man + woman = queen" применительно к вашей же истории с race condition: у вас есть **до и после** фикса (recovery-path без лока → с локом в `remote_embedder.py`). Возьмите эмбеддинг чанка до фикса и после, вычислите вектор-разницу `Δ = embed(fixed) - embed(buggy)` — это "направление исправления race condition" в семантическом пространстве. Дальше прогоняете **весь остальной граф кода** и ищете чанки, чьё положение относительно ближайшего "здорового" соседа похоже смещено в направлении, противоположном `Δ`. Гипотеза: это найдёт другие места в кодовой базе с похожим латентным паттерном (общий mutable-объект + lock, который где-то забыт) — то есть проактивно найдёт следующий race condition до того, как он проявится, а не после. Я не встречал ни одной публикации, где vector-arithmetic от одного known fix использовался бы как "детектор-щуп" по остальному коду — обычно semantic diff используют для описания изменения, а не для поиска изоморфных багов в других местах. У вас для этого эксперимента есть готовый, дешёвый, реальный тест-кейс уже сегодня.

## 2. Bitemporal Cypher — "запрос к графу кода, каким он был в коммите X"

Сейчас `PropertyGraph` хранит только текущее состояние. Добавьте на рёбра/узлы поля `valid_from_commit` / `valid_to_commit` (bitemporal модель, как в Datomic/XTDB, но для кода) и стройте их инкрементально из `git log --numstat` + tree-sitter diff по каждому коммиту, а не полным реиндексом. Тогда становится возможен запрос вида: *"каким был call-graph функции `embed_batch` на момент коммита `0665a4b`, до того как её вынесли в новый пакет?"* — без чекаута старого коммита и полной переиндексации. Это прямо решает вашу же прошлую боль ("перекачал main — а там опять другое") — можно диффать не файлы, а именно **структуру графа зависимостей** между двумя момента времени и увидеть, где реально сломался call-path, а не гадать по AGENT_DIARY. Инструментов, которые дают time-travel именно по графу символов (а не просто git blame по строкам), в open-source code-intelligence я не видел.

## 3. Sparse Autoencoder поверх ваших embeddings — интерпретируемые "оси смысла" вместо чёрного ящика

Из механистической интерпретируемости LLM (Anthropic же и публикует это направление) — SAE обучается находить в скрытом пространстве модели разреженные интерпретируемые "фичи". Никто пока не применял это к **embedding-пространству инструмента поиска по коду** конкретного проекта. Эксперимент: обучите небольшой SAE (это дёшево — эмбеддинги 384-dim, датасет — все чанки вашего же проекта) поверх выхода `multilingual-e5-small-int8`. Дальше можно вручную подписать найденные разреженные направления ("это ось async/await", "это ось error-handling", "это ось работы с файловой системой") и добавить режим поиска: *"покажи код рядом с осью 'concurrency-risk', не по текстовому запросу, а по обученному направлению"*. Это принципиально другой режим навигации, чем nearest-neighbor по запросу — по сути ваш собственный "code concept explorer", подстроенный конкретно под словарь понятий вашего проекта, а не универсальный.

## 4. Verification-cost-guided property-based fuzzing — превентивно, а не по факту

У вас уже есть статически детектируемый паттерн: `self._x = ...` + `self._x_lock = threading.Lock()` рядом. Cypher-запрос по вашему же графу может найти **все** такие пары атрибутов в кодовой базе (это буквально паттерн-матчинг по AST-узлам класса). Для каждой найденной пары — автогенерация конкурентного stress-теста через `hypothesis`/`pytest-asyncio` (N потоков одновременно дёргают метод, владеющий этим атрибутом, замер на невалидные состояния) — по сути то, что вы вручную сделали руками для одного конкретного `InferRequest` на этой неделе, но примененное сразу ко всем похожим местам в проекте автоматически, до того как баг проявится в проде. Насколько знаю, автогенерация race-condition фаззинга **из графа кода** (а не из ручного анализа конкретного модуля) в open-source тулинге не встречается — обычно fuzzing нацелен на inputs функций, а не на паттерн "shared mutable state + lock", извлечённый структурно.

## 5. Cascade retrieval с адаптивным откатом по точности (а не по фиксированному правилу)

Прямое продолжение вашей же истории с INT8/FP32: вместо жёсткого "сначала INT8, при ошибке — FP32" (текущая архитектура, которая и порождала весь этот баттл) — сделать откат **по confidence самого результата поиска**, а не по типу исключения. Идея: считать top-K через быстрый INT8, затем для этих top-K пересчитать через FP32 только если разброс similarity-скоров внутри top-K подозрительно плоский (recall низкой уверенности — типичный симптом деградированных/испорченных векторов, который у вас реально наблюдался как "результаты не отличимы друг от друга"). Это превращает "ловлю нулевых векторов постфактум логом" в "систему, которая сама замечает, что её же собственные векторы выглядят подозрительно, и перепроверяет себя" — self-diagnosing retrieval pipeline. В продакшн-поиске такое иногда делают для relevance (query rewriting при низком confidence), но не для детекции **деградации самого embedding-рантайма** как признака технической неисправности, а не плохого запроса.

## Быстрые победы (дни, не недели)

**1. Chunk-level content-addressed кэш вместо file-level.** Сейчас в `indexer.py` (`_calculate_file_hash`, строка 272) хеш считается **на весь файл** — поменяли одну функцию в файле на 2000 строк → перегенерируется весь файл целиком. Cursor (проверил их же техническую документацию) хеширует **каждый чанк отдельно** (SHA256 по содержимому чанка после tree-sitter split) и пропускает эмбеддинг для чанков с неизменным хешем, даже если файл вокруг них поменялся. У вас уже есть SHA256-инфраструктура и AST-парсинг — это не новая система, а расширение существующей на один уровень глубже. При типичном паттерне правок (1 функция в файле) это экономит 90%+ повторных эмбеддингов на реиндексе.

**2. Профилировать, а не гадать.** Пять дней потратили на теории (race condition, token_type_ids, batch size) вместо `py-spy record --pid <pid> --output profile.svg` во время реального инференса. Flame graph за 10 минут показал бы, тратится ли время в `.infer()` или в Python-обвязке (сериализация тензоров, GIL, копирование numpy-массивов) — возможно, часть "медленности" вообще не в OpenVINO. Бюджет: один вечер, `pip install py-spy`.

**3. `AsyncInferQueue` вместо lock+single request** (уже обсуждали) — это единственный пункт из прошлого аудита, который остаётся архитектурным долгом.

## Средний бюджет (неделя-две)

**4. Matryoshka-truncation для LanceDB, НЕ для скорости эмбеддинга.** Важная оговорка, которую нашёл прямо в документации sentence-transformers: Matryoshka **не ускоряет сам инференс модели** — она ускоряет то, что происходит после (ANN-поиск, хранение). Если `multilingual-e5-small-int8` — не Matryoshka-модель изначально, придётся её дообучить (дёшево, на своих же данных, MatryoshkaLoss). Даёт: обрезка векторов до 128-256 dim при поиске первого прохода (LanceDB IVF-índекс меньше, быстрее ANN), полный вектор — только для финального re-rank топ-20. Не решает вашу боль с эмбеддингом чанков при индексации, зато ускоряет запросы на чтение в 2-5 раз почти бесплатно по качеству (по бенчмаркам — >88% recall на 256 dim от 384).

**5. NNCF static quantization с сохранением dynamic batch axis** — устраняет корневую причину, почему у вас batch=1 захардкожен (`Multiply` node не поддерживает broadcasting на batch>1 в текущей квантизации). Если это исправить — реальный батчинг вместо последовательного цикла с локом на каждый чанк, это может дать больше прироста, чем весь пул/лок рефакторинг вместе взятый.

**6. Инкрементальный git-hook индексер по staged diff.** Сейчас реиндекс, похоже, идёт по всему проекту/файлу при триггере. `git diff --cached --unified=0` даёт точные изменённые строки → маппинг на AST-узлы tree-sitter → эмбеддится **только затронутый chunk**, ещё до полного скана. Это тот же принцип, что и п.1, но триггер — pre-commit hook, а не периодический скан. Дешёво: скрипт на 50-100 строк поверх уже существующего парсера.

**7. Shadow canary перед сменой embedder-модели** (обсуждали) — конкретно: 20-30 known query→chunk пар как канареечный набор, сравнение cosine similarity старой/новой модели перед переключением дефолта в `install.py`.

## Исследовательские, специфичные именно для вас (пробуйте на своих данных, риск выше)

**8. Vector-arithmetic propagation от известного фикса** (обсуждали) — `Δ = embed(after_fix) - embed(before_fix)` на реальном diff race-condition фикса → искать другие места в графе, смещённые в противоположном направлении.

**9. `verify_diary.py`** — парсит `AGENT_DIARY.md`, вытаскивает "✅ done"-утверждения, генерирует проверяемые условия (grep на класс/функцию, тест), гоняет на каждом старте. Это то, что реально устранило бы 4 из 6 расхождений, которые я нашёл вручную за этот чат.

**10. Bitemporal Cypher** (обсуждали) — `valid_from_commit`/`valid_to_commit` на рёбрах графа, инкрементально из `git log --numstat`, без полного реиндекса на исторический запрос.

**11. Auto-fuzzing по AST-паттерну "shared mutable + lock рядом".** Cypher-запрос по вашему же графу находит все пары `self._x` + `self._x_lock` → авто-генерация `hypothesis`/`pytest-asyncio` конкурентного стресс-теста для каждой найденной пары. То, что вы вручную сделали для одного `InferRequest`, но структурно найдено графом заранее, до прода.

**12. Simhash "мгновенный старт" для похожих проектов** (подсмотрел у Cursor, у вас нет аналога). Если человек открывает форк/похожий проект (например, тот же `mscodebase-intelligence` на другой машине, или ваш `gemma_agent`), считать simhash от Merkle-дерева файловых хешей и искать похожий уже готовый индекс локально в `~/.cache/mscodebase/` — если похожесть высокая, копировать существующий индекс как стартовую точку вместо полного реиндекса с нуля. У вас уже есть multi-window `ProjectIndexerRegistry` с LRU(5) — это расширение той же идеи на "холодный старт похожего проекта".

**13. SAE поверх собственных эмбеддингов** (обсуждали) — дёшево именно потому, что датасет — это чанки вашего же проекта, не нужен внешний корпус.

**14. Verification-cost score в Explainability-слое** (обсуждали) — не confidence, а "сколько файлов/минут нужно проверить, чтобы доверять этому ответу".

**15. Cascade retrieval с self-diagnosing confidence** (обсуждали) — откат INT8→FP32 по плоскому распределению similarity в top-K, а не по типу исключения; превращает баг с нулевыми векторами в самодиагностику, а не в постфактум-лог.

### 1. Архитектурный эталон: Swimm.io

Это проект, который сейчас ближе всего к твоей задаче. Они решают проблему "sync" на уровне IDE.

* **Как это работает:** Ты создаешь документ в IDE, который привязан к конкретным строкам кода (через AST).
* **Их "киллер-фича":** **Continuous Integration для документации.** Когда ты делаешь `git commit`, Swimm проверяет: «Сдвинулись ли строки кода, на которые ссылается документация?». Если да — документация помечается как `outdated` (грязная), и CI-пайплайн может заблокировать PR.
* **Что перенять:** Тебе не нужно "писать" документацию. Твой `PropertyGraph` уже знает, где находится `func_a`. Если код функции изменился (Content Hash изменился), ты можешь автоматически помечать любую документацию, ссылающуюся на эту функцию, как **"Stale / Needs Review"**.

### 2. Подход "Docs-as-Graph": Backstage (от Spotify)

Backstage — это стандарт для архитектурных каталогов в крупных компаниях.

* **Суть:** Документация — это не текст, а метаданные. Каждый сервис/функция описывается файлом `catalog-info.yaml`, который лежит рядом с кодом.
* **Почему это работает:** Автоматизация. Ты не пишешь описание руками, ты описываешь структуру (кто владелец, какой API, какие зависимости), а Backstage *генерирует* портал на основе этих метаданных.
* **Что перенять:** Твой `PropertyGraph` уже содержит `DEFINES` и `CALLS` ребра. Тебе нужно создать "View" (проекцию) этого графа. Вместо написания `README.md`, ты генерируешь `Graph View` функции, который всегда актуален, так как он берется из индекса.

### 3. API-First подходы: Swagger / OpenAPI / Storybook

Это самые старые, но рабочие примеры "автодокументации".

* **Принцип:** Документация — это "контракт". Если код не соответствует контракту (Swagger-спеке), билд падает.
* **Что перенять:** Ты можешь использовать `tree-sitter` для генерации "подписей" (signatures) функций. Если сигнатура меняется, ты можешь автоматически обновлять Markdown-файл с API-документацией проекта.

### Твой путь: как превратить `MSCodeBase` в движок "Living Docs"

У тебя уже есть всё необходимое (AST + PropertyGraph). Тебе не нужно искать готовый продукт, тебе нужно достроить **"Generator Layer"**.

#### Шаг А: Слой генерации (Projections)

Вместо ручного написания документов, реализуй генератор `README.md` из твоего PropertyGraph.

* Используй свой `PropertyGraph` для поиска `CALLERS` и `DEFINES`.
* Генерируй Markdown-блоками:
```markdown
## Функция process_data
* **Caller:** run_pipeline (в consumer.py)
* **Dependencies:** helper_func

```


* Этот Markdown будет **всегда актуален**, потому что ты генерируешь его из графа перед каждым коммитом.

#### Шаг Б: Детектор устаревания (The "Dirty" Flag)

Поскольку ты уже реализовал `Content Hash` (SHA256) в своем кэше:

1. Если `Content Hash` функции в файле изменился, а README файл не перегенерировался — помечай README как "устаревший".
2. Можешь выводить в Zed (или консоль): `⚠️ Warning: Documentation for process_data is out of sync with code.`

#### Шаг В: AI-Augmented Docs

Используй LLM не для *написания* документации, а для *описания* того, что ты уже вытащил через AST.

1. Ты парсишь `func_a`.
2. Отправляешь LLM: "Напиши docstring для функции, которая делает X, вызывается из Y и использует Z".
3. Записываешь результат в `PropertyGraph` (в таблицу `nodes` → столбец `docstring`).

### Резюме: от чего предостеречься

1. **Не храни текст документации отдельно от кода.** Она протухнет через 2 недели.
2. **Не пытайся сделать документацию "прозой".** Делай её структурированной (графы, списки параметров, зависимости). Прозу люди не читают, а код постоянно меняется.
3. **Используй "Stale-ness".** Самое ценное в живой документации — это знать, *какая её часть устарела*.
