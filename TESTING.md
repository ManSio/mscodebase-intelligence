# 📑 МЕТОДОЛОГИЯ И СЦЕНАРИИ ТЕСТИРОВАНИЯ (QA)

Эти сценарии используются для проведения регрессионного и нагрузочного тестирования при изменении кодовой базы расширения (например, при доработке `watcher.py` или `indexer.py`).

### Сценарий 1: Валидация нормализации путей (Path Validation Test)

* **Цель:** Проверить, что расширение всегда генерирует один и тот же статичный хеш для одного и того же проекта, независимо от формата путей Windows.
* **Шаги верификации:**
1. Инициализировать индекс для тестового пути вида `D:\Work\Project_Beta`.
2. Повторно запросить обращение к индексу, передав путь в альтернативном формате (сменив регистр диска `d:\...` или заменив обратные слэши на прямые `d:/work/project_beta`).

* **Ожидаемый результат:** Функция `_generate_unique_db_path` должна вернуть строго идентичный путь к базе данных: `<PARENT_DIR>/.codebase_indices/lancedb_v2/index_Project_Beta_a3019552.db`. Дублирующие папки баз создаваться не должны.

### Сценарий 2: Изоляция контекста при переключении окон (Context Switch Test)

* **Цель:** Убедиться, что при смене фокуса между активными окнами проектов в Zed расширение корректно изолирует запросы.
* **Шаги верификации:**
1. Эмулировать переключение рабочей среды с `Project_Alpha` на `Project_Beta`.
2. Выполнить вызов инструмента сбора контекста (`get_context`).

* **Ожидаемый результат:** Поток векторов перенаправляется строго в целевой файл базы данных текущего активного проекта. Данные из базы соседа (138 MB) не подмешиваются в итоговое контекстное окно модели.

### Сценарий 3: Инкрементальное обновление через Watcher (File Monitor Test)

* **Цель:** Проверить корректность работы инкрементального обновления индекса при изменении файлов в реальном времени.
* **Шаги верификации:**
1. Внести изменения в одиночный файл внутри `Project_Beta` (база 1.6 MB).
2. Перехватить событие файлового монитора (`on_modified` / `on_created`).

* **Ожидаемый результат:**
   - Расширение должно запустить процедуру векторизации *только* для измененного файла.
   - Старые векторы данного файла в LanceDB перезаписываются по его относительному ключу `file_path`.
   - Операция должна происходить изолированно за доли секунды, без повторного сканирования соседнего проекта `Project_Alpha` весом 138 MB.


### Сценарий 4: Тест поочередного сброса и удержания RAM (Memory Sequence Test)

* **Цель:** Проверить корректность освобождения памяти при поочередном открытии проектов в одном окне на тестовом стенде (16 GB RAM).
* **Шаги верификации:**
1. Открыть крупный проект (`Project_Alpha`), запустить полную переиндексацию, зафиксировать пик утилизации RAM (~1100 MB).
2. В рамках того же окна последовательно закрыть текущий проект и открыть `Project_Beta`.
3. Снять показания Диспетчера задач Windows для активных процессов Python.

* **Ожидаемый результат:** Старый процесс с накопленным кэшем в 1.1 ГБ должен полностью завершиться операционной системой, а новый инициализированный процесс для `Project_Beta` должен занять строго базовую позицию в пределах **~117 MB – 118 MB RAM**. Это подтверждает идеальный менеджмент ресурсов со стороны Zed и отсутствие скрытых утечек памяти.
